## Why

- The run-catchup.sh script was a thin shell stub whose sole function was to resolve the repo root and exec build-catchup.py. Once build-catchup.py gained --repo-root and --trigger flags the wrapper added no meaningful logic — it only introduced a file that had to stay in sync with the central install and an extra exec layer in every hook invocation. Removing it ensures hooks always use the canonical central-install Python script and makes the catch-up trigger path transparent and consistent with how other hook scripts already operate.

## What changed

- The catch-up trigger path was collapsed from a two-step shell-then-Python execution into a direct Python call. The shell wrapper that bridged the gap during early development was removed once it was confirmed to contain no logic beyond exporting env vars and exec-ing the Python script. The hook generation template was updated so all repos wired after this change produce hooks that call the central install Python script directly with explicit named arguments. The system design documentation was brought into alignment with the new single-step invocation model.
