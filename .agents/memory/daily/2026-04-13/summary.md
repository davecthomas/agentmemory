## Why this mattered

- The run-catchup.sh script was a thin shell stub whose sole function was to resolve the repo root and exec build-catchup.py. Once build-catchup.py gained --repo-root and --trigger flags the wrapper added no meaningful logic — it only introduced a file that had to stay in sync with the central install and an extra exec layer in every hook invocation. Removing it ensures hooks always use the canonical central-install Python script and makes the catch-up trigger path transparent and consistent with how other hook scripts already operate.
- The system must prevent raw, mechanical, single-turn data from becoming durable shared memory. A single agent turn produces an incomplete picture of the workstream; directly publishing it would fill the memory layer with noisy, context-poor entries that mislead future agents. By separating raw capture (always local-only) from durable publication (synthesized from a bounded episode cluster after trust validation), the system ensures that durable memory reflects coherent workstream intent rather than individual turn artifacts.
- When a wired repo has no event shards yet, the system needs to seed initial memory from existing commit history and design docs. The obvious path — injecting a "please bootstrap" instruction into the main agent's context — fails reliably in practice: the agent already has ADR index content loaded from `SessionStart`, sees that decisions are recorded, and reasons (incorrectly) that the repo is already initialized. It ignores the bootstrap instruction entirely. An isolated subagent spawned as a detached subprocess has no conversation history and no competing context; it receives only the `memory-bootstrap` skill as its system prompt and executes the task cleanly.
- The old agent/provider naming was ambiguous — it conflated the concept of an AI agent runtime (claude, gemini) with non-agent callers like git hooks, bootstrap scripts, and the installer. Consistently using runtime-id eliminates that ambiguity so log lines always identify the exact execution path that produced them, whether that is an agent turn, a git hook invocation, or a CLI script run. This makes post-mortem log analysis and hook-trace review reliable.
- The agentmemory hook output stream is shared by interactive AI agent sessions, git lifecycle hooks, and install scripts. Using [agent=...] and [provider-version=...] vocabulary was correct only for interactive agent invocations — git hooks and scripts have no 'agent' or 'provider version'. Normalizing to [runtime=...]/[runtime-version=...] makes log lines self-describing across every execution context, improving mixed-runtime debugging from a single trace file.
- The v0.4.3 logging normalization pass established a structured [agentmemory][version=...][runtime=...][runtime-version=...] prefix for all helper output. However, the generated post-checkout/post-merge/post-rewrite hooks still routed through scripts/shared-repo-memory/run-catchup.sh — a repo-local wrapper that added an indirection layer and required the repo-local scripts directory to be present at hook fire time. With run-catchup.sh deleted, all four generated hook scripts now invoke their helper (build-catchup.py or pre-commit-memory-guard.py) directly from the central install at $HOME/.agent/shared-repo-memory/, and every hook sets explicit runtime identity env vars before execution, completing the uniform identity scheme across agent sessions, git operations, and installer flows.
- The old checkpoint clustering mechanism grouped pending captures by thread ID or branch — a heuristic that could bundle unrelated work done on the same thread, or fragment related work done across threads. A deterministic local episode graph addresses this by building explicit associations from repo-grounded signals: shared file paths, adjacent timestamps, and matching workstream IDs. Clustering from graph traversal produces episode boundaries that reflect actual semantic relationships rather than incidental provenance metadata. Persisting per-episode manifests under `.agents/memory/state/episode-graph/episodes/` makes the clustering logic locally inspectable without requiring a graph library or external service.
- Claude Code's compaction (context window summarization) discards the full transcript, including the ADR index and daily summaries injected by the `SessionStart` hook. Without a recovery mechanism, the agent loses awareness of past decisions and recent work for the remainder of the session — exactly the context that `SessionStart` was designed to provide. The `PostCompact` hook re-injects the same bounded read set so the memory system's core invariant holds even after mid-session compaction: every agent turn, whenever it occurs in a session, starts from prior decisions rather than rebuilding history from scratch.
- The previous logging schema used 'agent' and 'provider-version' fields that implied every agentmemory caller is an AI agent. In practice, git hooks, the installer, and manual CLI invocations are not agents and were logging as 'unknown', making hook-trace records ambiguous when debugging mixed-runtime workflows. A runtime-neutral schema lets every caller self-identify accurately.
- Context window compaction is a runtime lifecycle event that discards the full transcript, including the memory context injected by `SessionStart`. Without explicit handling, the agent loses awareness of ADRs and recent daily summaries for the remainder of the session — all the context that `SessionStart` carefully loaded is silently gone. The `PostCompact` hook exists solely to re-inject the same bounded read set (ADR index + three most recent daily summaries) so the agent continues to operate with the same ground truth after compaction as before.

## Active blockers

- Local catch-up (`catchup.md`) is an uncommitted digest that reflects what changed since the last rebuild. It is not shared memory (never committed) and cannot be refreshed from the network. Without an automatic rebuild trigger, a `git pull` or branch switch would leave the catch-up stale: the agent's next session would read outdated "what's new since your last session" context even though the working tree already contains the updated shared-memory files. Using native git hooks — specifically `post-checkout`, `post-merge`, and `post-rewrite` — means the rebuild fires at exactly the right moment (after the tree changes) with no extra developer action and no polling loop.
